Draft reports

These worked examples show how an AI agent can turn public aggregate NHS data into cautious, reviewable first-draft briefs. Each follows a governed workflow — question, data, analysis, caveats, human review — and is not official Dorset HealthCare performance reporting.

Demonstration caveat: These are agent-assisted analytical briefs using public aggregate data only. They are not official Dorset HealthCare reports. They require human review, subject matter input and local owner confirmation before any operational use.

Public-data agent workflow examples

"Aggregate public data" means published NHS figures that are already grouped to organisation level — counts and rates, never individual patients. A "first-draft brief" is exactly that: a starting point an analyst would refine, not a finished report. Drafting still needs a person because definitions vary, data can be provisional or incomplete, and figures only mean something once they are read alongside local context and judgement.

Each brief below demonstrates how a governed agent interprets a business question, locates public aggregate data for Dorset HealthCare — identified by its Organisation Data Service (ODS) code RDY — produces a cautious first-draft analysis, and surfaces what a human reviewer must still check. The workflow is always the same: question → data → analysis → caveats → human review. See also the agent operating model.

Reports are rendered from public-data/processed/demo_*.csv by R/03_render_public_reports.R. Source catalogue: DATA_SOURCE_REGISTER.csv.

How these briefs were created

  • A Cursor agent discovered and downloaded official public aggregate NHS sources — catalogued in DATA_SOURCE_REGISTER.csv.
  • R scripts preserved raw files, inspected fields, filtered RDY/Dorset HealthCare rows, and created public-data/processed/demo_*.csv extracts.
  • R/03_render_public_reports.R renders each brief using a shared agent-assisted analytical template: question, prompt excerpt, process steps, Key figures explained, first-draft analysis, figures, caveats and verification appendix.
  • Agentic AI accelerated discovery, drafting and checking; R produces reproducible tables and charts from bounded public data only.
  • Every brief ends with explicit non-claims and a human review gate — no operational use without local owner sign-off.

Supporting documentation:

  • PUBLIC_REPORTS_METHOD.md — pipeline and agent brief template
  • AGENT_ANALYTICAL_BRIEF_REFRAME_PLAN.md — reframe specification
  • FINAL_REPORT_QA_SUMMARY.md — final report clarity/trend QA pass
  • PUBLIC_DATA_RUN_SUMMARY.md — run summary and source status
  • DATA_SOURCE_REGISTER.csv — public source catalogue

Regenerate public reports with: Rscript site/R/03_render_public_reports.R (after running the public-data pipeline).

Why this matters for the Business & Performance Business Partner role

Read as examples rather than claims, these briefs show practical skills the business-partner role relies on:

  • Turning a question into a reviewable first draft — each brief shows the question asked, the process followed, and the output a reviewer would receive.
  • Interpreting public datasets cautiously — suppression, provisional data, comparator limits and explicit non-claims are surfaced rather than hidden.
  • Spotting data quality and definitional risks — organisation matching, source availability and metric directionality are flagged before any narrative is written.
  • Communicating analysis clearly to non-analysts — a consistent structure makes complex performance information readable for services and senior leaders.
  • Using agentic AI within governance boundaries — AI accelerates discovery, drafting and checking, but humans stay accountable: no brief is presented as approved, and local owner sign-off is required before operational use.
